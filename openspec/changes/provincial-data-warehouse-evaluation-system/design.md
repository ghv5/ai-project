## Context

**当前状态**：
- 项目为全新系统，无现有代码
- 数仓和规则为黑盒，通过预配置数仓列表接入
- 无需考虑用户权限和认证

**约束条件**：
- 数仓列表预配置，不从外部系统动态获取
- 大屏需实时更新数据
- 报告分享支持直接查看，无需登录
- 无需主导航栏结构

## Goals / Non-Goals

**Goals:**
- 实现三大功能模块：单个评估、评估报告、省级大屏
- 提供完整的页面布局和页面元素设计
- 大屏支持实时数据刷新
- 报告支持 PDF/Word 导出

**Non-Goals:**
- 用户权限管理系统
- 数仓和规则引擎的实现
- 多租户支持
- 移动适配（默认桌面端优先）

## Decisions

### Decision 1: 页面路由结构
**选择**：扁平化路由结构
```
/evaluations          - 评估列表
/evaluations/new      - 新建评估
/evaluations/:id      - 评估详情
/reports              - 报告列表
/reports/:id          - 报告预览
/reports/generate     - 报告生成配置
/dashboard            - 省级大屏
```
**理由**：功能模块清晰，无需嵌套路由。

**替代方案**：按模块分组路由 `/module/feature`，但增加了复杂度。

---

### Decision 2: 大屏实时更新方案
**选择**：WebSocket 推送 + 本地轮询降级
- 优先 WebSocket 接收服务端推送
- WebSocket 断开时自动降级为定时轮询
- 支持 5s/10s/30s 可配置刷新间隔

**理由**：兼顾实时性和可靠性。

**替代方案**：仅轮询（延迟高）、仅 WebSocket（可靠性差）。

---

### Decision 3: 报告生成方案
**选择**：前端驱动 + 后端渲染
- 前端收集配置参数
- 后端基于模板生成 PDF/Word
- 生成过程中前端轮询状态

**理由**：利用后端成熟的文档生成库。

**替代方案**：纯前端生成（大型报告性能差）。

---

### Decision 4: 数据存储（黑盒接口定义）
**选择**：定义标准化接口契约
- 评估启动接口：POST /api/evaluations
- 评估详情接口：GET /api/evaluations/:id
- 评估结果导出：GET /api/evaluations/:id/export
- 报告生成接口：POST /api/reports
- 大屏数据接口：GET /api/dashboard/metrics (WebSocket 或 REST)

**理由**：与黑盒系统解耦，便于后续替换。

---

### Decision 5: 大屏统计口径（2026-03-16 Research）
**选择**：按“部门-数仓”作为领导视角主维度，且统一使用“最新一次评估”口径。
- 统计主维度为部门，不以地市为主展示维度
- 同一数仓多次评估时，只取最新一次评估参与大屏核心统计
- 待改进数按最新评估中 `grade in C/D` 统计
- 大屏核心补充“已评估覆盖率”
- 报告相关仅展示“报告生成数量/覆盖率”
- 报告覆盖率分母为“已完成评估数”
- 大屏不使用列表扩展 mock（不引入额外虚拟数仓）
- 趋势图采用“每月末最新状态快照”口径
- 图表库暂不做预设（实现阶段再按工程约束选型）

**理由**：符合省级领导看板决策场景，避免多次评估重复计数造成失真。

## 信息架构（Plan）

### 首屏分区

1. 总览指标区：部门数、数仓数、已评估覆盖率、平均得分、优秀率、待改进数、报告覆盖率
2. 结果分布区：优秀数仓 Top 5、数仓得分分布图、数仓等级分布饼图
3. 根因分析区：规则集维度雷达图、规则引用数 Top 5、高频问题清单

### 布局决策（2026-03-16 Research）

**选择**：三层信息布局，自上而下从结果到原因。
- 第一层只放总体指标，不放操作控件和次要统计
- 第二层回答“哪些数仓表现最好、整体分数如何分层”
- 第三层回答“问题主要集中在哪些规则和高频问题”
- 不再保留部门热力图和月度趋势图作为首屏主模块

**理由**：更符合省级领导的阅读习惯，首屏先看质量结论，再看结构分布，最后看问题根因。

### 指标公式（冻结）

- 最新评估记录（按数仓）：`latestEval(warehouse) = max(evaluationTime)`
- 待改进数：`count(latestEval.grade in ['C','D'])`
- 已评估覆盖率：`count(has latestEval) / count(all warehouses)`
- 报告生成数量：`count(all reports in time range)`
- 报告覆盖率：`count(reports linked to completed evals) / count(completed evals)`
- 趋势（月）：`avg(score of monthEndLatestEval by warehouse)`
- 规则引用数 Top 5：`count(issue.rule in latestEval issues)`
- 责任部门：某问题出现次数最多的部门

### 模块拆分（Implementation Plan）

1. 数据适配层
- 输入：`warehouses`、`evaluations`、`reports`、`dimensions`
- 输出：统一大屏视图模型 `dashboardViewModel`
- 责任：按最新评估口径聚合所有指标、榜单、分布、饼图、雷达图和问题清单

2. 指标卡区域
- 展示：部门数、数仓数、已评估覆盖率、平均得分、优秀率、待改进数、报告覆盖率
- 目的：让领导先看规模、质量、闭环

3. 结果分布区域
- 模块 A：优秀数仓 Top 5
- 模块 B：数仓得分分布图
- 模块 C：数仓等级分布饼图
- 目的：让领导快速识别质量头部与整体分层

4. 根因分析区域
- 模块 A：规则集维度雷达图
- 模块 B：规则引用数 Top 5
- 模块 C：高频问题清单
- 目的：从维度、规则、问题三个层级定位质量短板

5. 页面控制区
- 展示：自动刷新、刷新间隔、主题切换
- 目的：控制刷新节奏，不改变统计口径

### 实施顺序

1. 完成数据适配层和口径单测/样例校验
2. 完成静态布局骨架与 3 层页面结构
3. 接入总体指标与结果分布区
4. 接入规则雷达、规则引用、高频问题
5. 接入自动刷新和主题切换
6. 逐项对照验收标准验证

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| 大屏 WebSocket 连接不稳定 | 提供轮询降级方案，连接失败自动切换 |
| 报告生成耗时过长 | 异步生成 + 进度反馈，支持取消 |
| 数仓黑盒接口变更 | 定义适配器层，隔离变化 |
| 大屏数据量过大渲染慢 | 数据聚合 + 虚拟滚动 + 防抖更新 |
