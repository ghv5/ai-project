## Context

### Current Local State

基于本地代码和现有文档，当前状态如下：

- 菜单中已新增“数据源”入口，但页面仅展示 3 个静态统计卡片
  - `src/js/app.js`
- 当前核心数据模型仅有：
  - `warehouses`：按地市 -> 部门 -> 数仓嵌套定义
  - `evaluations`
  - `reports`
  - `ruleSets` / `rules`
- 当前不存在：
  - 数据源实体
  - 连通性测试结果实体
  - 数仓创建向导或建仓状态
  - 数仓与部门映射维护页
- 现有 OpenSpec 中明确声明：
  - “数仓列表预配置，不从外部系统动态获取”
  - 这与本次需求存在冲突

### Requirement Shape

用户需求原文：

> 这个页面功能为数仓的结构化数据源接入并管理，可以新建数据源，测试连通性、数仓创建、数仓以及部门关联

这意味着“数据源页面”至少不是一个纯展示页，而是一个具备管理闭环的业务页面。

### Confirmed Decisions (2026-03-17)

- “数仓创建”定义为：创建逻辑数仓对象，字段可参考现有数仓列表卡片所需信息
- 一个数仓只能关联一个部门
- 数据源最小字段集固定为：
  - 名称
  - 类型
  - 主机
  - 端口
  - 数据库名
  - 用户名
  - 密码
- 连通性测试结果仅即时查看，不保留历史日志
- 数仓在完成部门关联前，不允许出现在可评估清单中
- 部门清单单独维护，但展示口径需与数仓列表中的部门一致
- 连通性测试采用同步返回
- 逻辑数仓对象除现有数仓信息外，额外维护“连通状态”
- 数据源与数仓关系固定为一对一
- 数据源只存在于“创建数仓”流程中，其他业务场景均以数仓为主对象
- 部门清单为内置配置数据，不在页面上提供增删改查
- 数仓创建完成后，需要在数仓详情中查看对应的数据源配置摘要

## Goals / Non-Goals

### Goals

- 定义数据源页面的最小业务闭环
- 明确“数据源 -> 数仓 -> 部门”的关系边界
- 定义可验证的前端交互状态
- 为后续 `/prompts:gudaspec-plan` 输出可拆分任务

### Non-Goals

- 不定义真实数据库 DDL
- 不实现后端驱动、驱动包安装或真实连通性探测协议
- 不在本阶段决定具体协议差异（MySQL / PostgreSQL / Hive / Oracle 等）

## Context Boundaries

### Boundary A: 数据源页面与页面信息架构

当前页面仅是占位内容，因此新的信息架构至少需要覆盖：

- 创建数仓入口
- 数据源配置表单
- 连通性测试反馈
- 逻辑数仓创建结果
- 数仓与部门关联维护区

已确认不需要长期维持“数据源管理列表”作为独立主对象视图，因为数据源只在创建数仓过程中存在。

### Boundary B: 前端领域模型

当前 `API.warehouses` 只表达“地市/部门/数仓”的静态嵌套，不表达来源。

本次需求至少需要新增以下领域对象中的一种实现方式：

1. `dataSources`
2. `warehouseCatalog`
3. `warehouseDepartmentBindings`
4. `connectivityTestRecords`
5. `departments`

若不拆分这些对象，后续会把“连接配置、接入状态、部门归属、评估对象”混在同一个数仓结构里，导致页面逻辑无法收敛。

其中已确认：

- `departments` 作为单独维护的配置清单存在
- `warehouseCatalog` 至少需要扩展 `connectivityStatus`
- `dataSources` 更适合作为建仓流程中的临时或过程性实体，而不是长期主列表实体

### Boundary C: 与数仓列表/评估的联动

现有系统里，数仓列表和评估页都默认“数仓已经存在且可直接评估”。

如果引入数据源接入流程，需要至少明确：

- 未完成接入的数据源是否能生成数仓
- 未完成部门关联的数仓是否能出现在数仓列表
- 未完成接入或未关联部门的数仓是否允许发起评估

已确认：

- 可以在数据源上下文中创建逻辑数仓对象
- 未完成部门关联的数仓不进入可评估视图
- 一个数仓仅允许绑定一个部门
- 一个数据源仅对应一个逻辑数仓对象
- 创建完成后，后续浏览、评估、管理都以数仓对象为主

## Hard Constraints

- 必须基于当前本地代码判断，不能假设已有后端或真实数据源驱动
- 必须显式处理与“预配置数仓列表”这一既有约束的冲突
- 页面功能至少覆盖：
  - 新建数据源
  - 测试连通性
  - 数仓创建
  - 数仓与部门关联
- 不能把“连通性成功”直接等价为“数仓已创建”或“已完成关联”
- 页面需要可表达多个状态，而不是单一成功/失败
- 数据源表单字段最小集固定为：名称、类型、主机、端口、数据库名、用户名、密码
- 数仓与部门关联为一对一关系
- 未完成部门关联的数仓不得进入可评估状态
- 连通性测试结果仅支持即时查看，不要求历史留存
- 部门清单需要独立维护，不直接从当前数仓嵌套结构反推
- 连通性测试交互按同步返回设计
- 逻辑数仓对象需要额外表达“连通状态”
- 数据源与数仓之间为一对一关系
- 数据源不作为后续管理主视图对象长期存在
- 部门清单作为内置配置提供，只读使用
- 数仓详情需要提供对应的数据源配置摘要查看能力

## Soft Constraints

- 延续当前亮色后台风格与左侧固定菜单布局
- 优先复用现有卡片、表格、按钮交互模式
- 尽量采用分步或分区式结构，避免一个超长单表单
- 在前端 mock 阶段保留最小副作用，避免过早引入复杂假接口

## Proposed Interaction Model

建议把页面拆为两个主区域：

1. 创建数仓流程区
- 填写数据源配置
- 测试连通性
- 创建逻辑数仓对象

2. 数仓结果管理区
- 展示已创建数仓
- 展示连通状态
- 维护所属部门
- 仅支持一对一部门映射

## State Model

建议最少引入以下状态：

### 数据源状态

- `draft`：已创建但未测试
- `testing`：测试中
- `test_success`：测试成功
- `test_failed`：测试失败

说明：该状态仅在“创建数仓”流程上下文中存在，不作为长期运营对象状态。

### 数仓接入状态

- `uncreated`：未创建
- `creating`：创建中
- `created`：已创建
- `bind_pending`：待关联部门
- `ready`：可评估

## Dependencies

- 本地前端 mock API 层需要新增数据源相关实体
- 本地前端 mock API 层需要新增部门配置清单实体
- 数仓列表页面需要识别“来源于数据源接入”的数仓视图
- 评估流程需要明确仅对 `ready` 状态数仓开放
- 数仓详情页需要扩展数据源配置摘要展示区
- 既有 OpenSpec 变更需要更新“预配置接入”设计假设

## Risks

| Risk | Why it matters | Suggested mitigation |
|------|----------------|----------------------|
| 既有模型冲突 | 当前数仓模型没有来源概念 | 在设计中单独引入数据源实体和绑定实体 |
| 页面职责过载 | 新建、测试、建仓、关联都在同页可能过重 | 采用“创建流程区 + 数仓结果区”二段式结构 |
| 状态定义不清 | 测试成功、建仓成功、可评估并不是同一层状态 | 拆成“连接状态”和“接入状态”两条状态线 |
| 部门关联规则不清 | 一个数仓是否只能归属一个部门未明确 | 在 plan 前先锁定关联基数 |
| 与现有评估入口耦合 | 数仓列表默认都可评估 | 需要定义准入条件并在 UI 层禁用不满足对象 |

## Success Criteria

- 能清晰区分“数据源管理”和“数仓接入管理”两类操作
- 数据源可被创建，并能表达至少一次测试结果
- 数仓不能在未完成必要步骤前直接进入可评估状态
- 页面能明确展示数仓与部门的关联结果
- 设计文档显式修正“数仓仅预配置接入”的旧假设
- 一个数仓只能关联一个部门
- 未关联部门的数仓不能出现在可评估入口中
- 页面可从独立部门清单中完成部门选择
- 连通性测试可同步给出成功/失败结果
- 创建流程中一个数据源只能生成一个逻辑数仓
- 创建完成后，其他页面不再以数据源为主视图
- 部门清单作为内置配置被只读使用
- 数仓详情可查看对应的数据源配置摘要
